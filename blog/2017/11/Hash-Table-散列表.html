Hash Table 散列表

發表於 2017-11-06 | 分類於 Data Structures and Algorithms 數據結構與算法

散列函數

散列函數(Hash function)，又稱哈希函數，是從任意類型或類映射至一整數(散列值，Hash value)的函數。Java的類Object聲明瞭hashCode方法，默認基於對象地址，因此可能違背下文提到的確定性特性，故而hashCode須應需重寫。

字符串散列函數(並非理論上最緊湊的排布)：

長整型與雙精度浮點數散列函數：左右兩半取抑或(XOR)。

散列函數特性

確定性 Determinism

兩次相同輸入的散列函數應返回相同結果。如其本身並非輸入數據，函數不應基於時間(Execution time)、對象地址(Object address)、可以在不改變對象身份的前提下改變值的數據(Transient Data，如人的名字、商品的價格)等。

均勻性 Uniformity

對於滿足任意分佈的一組輸入，散列函數的輸出應均勻分佈於其值域(整數域)，使得相鄰輸入會被拉開。可以生成多組不同隨機分佈的輸入，對每組的輸出進行卡方檢驗(Chi square test)。本博客採用JUnit進行性能測驗。testUniformity中的refreshRange、renew等Hashable接口方法將會在下文實現。

抗碰撞性 Collision Resistance

當散列函數不是雙射時，會存在兩個不同的輸入被映射到同一散列值，稱爲碰撞(Collision)。這裏從效率角度來講，散列函數應該儘量避免碰撞。但事實上，抗碰撞性是個比均勻性更弱的需求。更均勻的輸出分佈意味着更少的碰撞。以下給出一個關聯性分析。

加密散列函數 Crytographic Hash Function

加密散列函數滿足基本的三條特性：

  1. 抗原象攻擊(Preimage Resistance)：對散列函數$h:X\rightarrow Y$，計算$h^{-1}$是困難的。
  2. 抗第二原象攻擊(Second Preimage Resistance)：已知$x \in X$，求$x’ \in X$，滿足$h(x’)=h(x)$是困難的。又稱弱抗碰撞性。
  3. 抗碰撞性：這裏從抗碰撞攻擊角度將，定義爲求$x, x’ \in X$, 滿足$h(x)=h(x’)$是困難的。又稱強抗碰撞性。

但仍然有諸多產生碰撞的方法，提供時間戳造假和散列碰撞拒絕服務攻擊(Hash Collision DoS Attack)。

時間戳(Timestamp)，即通過在某時間公佈某文檔的散列值，證明該文檔在該時間之前已經存在。由加密散列函數的抗原象攻擊性，很難在事後產生特定散列值的文檔。而且因爲文檔通常爲證書、知識產權等有意義內容，事後生成更加困難。

解決碰撞 Collisions Resolution

開放尋址法 Open Addressing

採用線性探測解決碰撞時，自發生碰撞的位置開始往後尋找空位，直到找到空位或已對數組遍歷一週(無法再找到空位)。事實上，由於密集的數組每次添加都可能需要大量的空位尋找操作(刪除操作則更是麻煩)，線性探測一般維護一數組的負載係數$\lambda$(Load factor)，當元素數目超過負載係數時，重新確定數組大小、散列函數並重新散列所有元素。

線性探測 Linear Probing

線性探測依照次序線性搜索空位。常用於實現集合(Set)。

二次探測 Quadratic Probing

拉鍊法 Seperate Chaining

採用拉鍊法解決碰撞時，容器數組不直接存放元素，而是碰撞元素的鏈表。常用於實現字典(Dictionary)。以下就分別以線性探測和拉鍊法實現集合與字典。

集合 Set

作爲最簡單的關聯(無序)容器(Associative container)，一個集合通常只實現判斷包含(contains)的訪問方法(Accessor)。而採用散列表實現可加速其搜索過程(不需要遍歷容器數組，只需基於散列值獲得存儲地址)。

字典 Dictionary

字典的核心訪問方法是搜索(通過鍵尋找值)。注意我們使用拉鍊法解決碰撞，在插入、計算地址時從整數域縮放到”[0, 數組長度)”，額外引入容器數組長度，會令碰撞的元素可能具有不同的鍵散列值(碰撞元素的鍵散列值可以是與其地址關於數組長度同餘的所有整數)，插入元素時根據鍵散列值選擇恰當位置，可進一步加速搜索過程。

可以觀察到，集合其實同構於的特殊字典。

集合與字典有時也用二叉查找樹(Binary Search Tree)實現(稱爲Tree Set或Tree Dictionary)。後者雖然正常添加、查找、刪除的效率($O(\mathrm{lg}n)$)不如散列表($O(n)$)，且相較兩種衝突解決方案都佔用額外空間；但卻避免了衝突解決帶來的如添加及查找時線性探測、遍歷衝突鏈表、刪除及增表時重新計算地址等的一系列開銷。當使用者對數據的量和分佈情況無法確定的時候，可以使用BST避免衝突解決風險。(注意：由於二叉查找樹不一定是完全二叉樹(Complete Binary Tree)，其如果用數組實現將會十分冗餘，故不予考慮)。
